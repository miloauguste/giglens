### 2. Town Estimation Overhaul — Pin-Based Bearing (v0.1.207 → v0.1.208)

**Problem:** Town estimates were wrong for every corrected order this session:

| DB ID | Restaurant | Estimated | Actual (user-corrected) |
|-------|-----------|-----------|------------------------|
| 8 | Burger King | Pemberton Township | Eastampton, NJ |
| 9 | Applebee's | Medford Township | Burlington, NJ |
| 15 | Wendy's | Delran Township | Burlington, NJ |

**Root cause — v0.1.207 (first attempt, wrong):**  
The estimator used `driverLocation.bearing` (device GPS compass) to project the delivery endpoint from the restaurant. But drivers are **parked and stationary** when offers arrive, so the device bearing is stale or 0.0 (north). A fix was attempted using the driver→restaurant coordinate bearing, but this was also wrong.

**Why coordinate bearing is also wrong (user clarified):**  
The customer is **not necessarily along the driver→restaurant direction**. The customer can be in any direction from the restaurant. The only reliable direction source is the DoorDash map itself — specifically the pixel positions of the pickup pin and dropoff pin.

**Fix — v0.1.208 (correct):**  
Replaced GPS/device bearing entirely. Delivery direction is now sourced from map pin pixel positions only.

**Three-tier estimation logic (`DeliveryTownEstimator.kt`):**

```
Priority 1 — Full pin detection (confidence: high)
  - Requires: driver dot + pickup briefcase pin + dropoff house pin
  - Method: PinDetectionTownEstimator (pixel calibration → delivery miles + pin bearing)
  - DB method field: "pin_detection"

Priority 2 — Partial pin detection (confidence: medium)
  - Requires: pickup + dropoff pins (no driver dot)
  - Method: pixel bearing from pickup→dropoff + GPS-computed delivery leg distance
  - DB method field: "partial_pin"

Priority 3 — No pins (confidence: low)
  - Result: TownEstimate(null, "low", "unavailable", "📍 ---")
  - NO projection is made — returning a wrong town is worse than returning nothing
  - DB method field: "unavailable"
```

**Critical behavior change:**  
`PinDetector.latestResult` is now always cleared at the top of `estimateTown()`, regardless of which path is taken. Previously it was only cleared inside the `success == true` branch, causing stale pin results from a prior offer to bleed into the next one.

**Files changed:**
- `geocoding/DeliveryTownEstimator.kt` — full rewrite of fallback path

### 3. Estimated Town in Pill Heading (v0.1.209 → v0.1.210)

**Request:** Show the estimated delivery town in the collapsed pill (heading), not just in the MINI/FULL drawer body. Town in heading is designated a **Pro (paid) feature**.

**Implementation (`OfferOverlayService.kt`):**

Pill PILL-state heading now shows:
```
+$5.50 45s · 📍 Burlington
```
- Net value + countdown: white, 14sp bold (existing)
- `· 📍 Burlington`: teal (`#00C9A7`), 80% relative size, same line
- Town stripped of `📍 ~` prefix for compact display
- Only shown when estimate is available (not `📍 ---`)
- Falls back to `+$5.50 45s` (no town) when estimate unavailable

**Method:** `pillTextWithTown()` — builds a `SpannableStringBuilder` with `RelativeSizeSpan(0.80f)` + `ForegroundColorSpan(#00C9A7)` on the town segment.

**Gate note:** Code comment marks this as a Pro feature. The gate logic (subscription check) has not been implemented yet — currently always shown when estimate is available.

**MINI and FULL drawer:** Already had town — no changes needed there.

**Files changed:**
- `service/OfferOverlayService.kt`

## Known Issues & Open Items

### High Priority

**1. Missing order #7 (7-Eleven Burlington)**  
GigLens was running but never captured this offer. Possible causes:
- DoorDash displayed the offer as a stacked/batched order (different UI than single-offer popup)
- Accessibility service missed the trigger keywords on that offer screen
- Offer came in as a notification-only ping rather than a full-screen popup
- Next step: check logcat from that shift window (~12:45–1:07 AM) for accessibility events

**2. Town estimation accuracy — all old rows use `gps_bearing`**  
IDs 1–9 and 15 in DB were estimated using the old broken bearing method. These town estimates are likely wrong. The new pin-based method (v0.1.208+) will produce better estimates going forward, but old rows are not retroactively fixable without re-running the offer through the estimator.

**3. OCR missing restaurant on IDs 12, 13, 14**  
Three consecutive offers in the same area (~12:28–12:43 AM) have no restaurant name. This could be:
- A DoorDash UI variant (stacked offers, different layout)
- OCR parsing failure on that screen format
- Accessibility service reading a partial/transition screen
- Next step: look for the debug email payloads from those captures in `getExternalFilesDir/debug/`

**4. `accepted` field always null**  
The DB has no way to distinguish offers the driver accepted from ones they declined. This matters for analytics accuracy. The `accepted` field exists in the schema but is never populated. Implementing accepted/declined detection would require either:
- Monitoring for the "order confirmed" / "heading to pickup" screen in the accessibility service
- A manual "I took this" button in the pill UI

**5. Town confirmation notifications — 0 taps recorded**  
`confirmedTown` and `townAccurate` are null on every row. The driver has not tapped Yes/No on any town accuracy notification. This data is critical for measuring whether the new pin-based estimation actually improves accuracy. Next shift: actively tap the notifications.

### Medium Priority

**6. South Star Diner duplicate (IDs 3 & 4)**  
Same pattern as Shak's Chicken — captured twice 3 minutes apart on 2026-06-18. Predates the dedup fix. Old data, not actionable now.

**7. Pro feature gate not implemented**  
Town in pill heading (`pillTextWithTown()`) is marked as a Pro feature in code comments but is currently shown to all users unconditionally. The subscription/Pro check needs to be wired in before public launch.

**8. `pinDetectionEnabled` flag**  
The pin detection path in `DeliveryTownEstimator` is gated by a DB flag (`AppConfigKeys.PIN_DETECTION_ENABLED`). This flag must be `true` for partial-pin and full-pin paths to run. Verify this is set correctly in Settings for the test device.
